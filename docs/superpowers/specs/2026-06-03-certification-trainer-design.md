# Тренажёр подготовки к сертификации системного аналитика IBS — Дизайн

**Дата:** 2026-06-03
**Статус:** утверждён к реализации
**Аудитория:** ~15 пользователей, production

## 1. Назначение

Web-приложение для тренировочного тестирования при подготовке к реальной
сертификации системного аналитика IBS. Генерирует тренировочные тесты через
LLM (OpenAI), имитирует условия экзамена (таймер, проходной балл), выдаёт
разбор ответов и персональные рекомендации по изучению тем.

Два уровня сертификации:

| Уровень | Вопросов | Время | Проходной балл |
|---|---|---|---|
| Базовый (`base`) | 80 | 120 мин | 70% |
| Специалист (`specialist`) | 120 | 180 мин | 75% |

## 2. Ключевые продуктовые решения

1. **Генерация на сессию.** Вопросы генерируются LLM при каждом запуске теста
   (не из заранее заготовленного банка). Время генерации не ограничено.
2. **Два режима теста:**
   - `exam` — честная симуляция: фиксированные официальные пропорции тем,
     полный объём (80/120 вопросов).
   - `adaptive` — тренировка слабых тем: короткий тест по темам, где пользователь
     проседает (по накопленному профилю компетенций).
3. **Адаптивность.** После каждого теста обновляется профиль компетенций по
   темам (`topic_competency`). В режиме `adaptive` генерация смещается на темы с
   наименьшей точностью. В режиме `exam` пропорции фиксированы (профиль всё равно
   обновляется).
4. **Только закрытые вопросы.** Типы `single` (один верный ответ) и `multi`
   (несколько верных). Никаких заданий, требующих внешних ресурсов — всё решается
   на странице. Проверка ответа детерминированная (сравнение с эталоном), без
   LLM-судьи в момент проверки.
5. **Артефакты в вопросах.** Код / JSON / SQL / XML — текст в `<pre>`. Диаграммы
   (UML class/sequence/state, ER, flowchart) — через **Mermaid** (LLM генерирует
   Mermaid-код как текст, браузер рендерит в SVG). Никаких сгенерированных
   картинок-диаграмм.
6. **Объяснения генерируются вместе с вопросом** (на этапе подготовки пула), а не
   в момент показа результатов — показываются мгновенно, без доп. вызовов.
7. **Финальная рекомендация** — один LLM-вызов в конце теста с учётом истории
   слабых тем.
8. **Качество вопросов — self-проверка.** Каждый сгенерированный вопрос проходит
   второй LLM-вызов (дешёвой моделью): проверка корректности, однозначности и
   верности эталона. `rejected` → перегенерация (макс. 2 повторные попытки на
   слот).
9. **Разделение моделей.** Генерация — сильная модель OpenAI (минимум
   галлюцинаций). Self-проверка — модель попроще (дешёвая работа). Конкретные
   модели — в конфиге; сверить с актуальным каталогом OpenAI при реализации.
10. **Регистрация открытая + защита.** Любой регистрируется сам (логин + пароль).
    Защита: (а) **код доступа** при регистрации (общий, сообщается своим людям),
    (б) **дневной rate-limit** на число запусков тестов на пользователя. Значение
    кода и лимит — конфиг.

## 3. Технологический стек

- **Frontend:** Next.js (React) — UI экзамена, рендеринг Mermaid и `<pre>`,
  таймер, навигация по вопросам, экран результатов.
- **Backend:** FastAPI (Python) — REST API + SSE, аутентификация, оркестрация
  генерации через OpenAI, оценка, профили, рекомендации.
- **БД:** PostgreSQL.
- **LLM:** OpenAI SDK (Python). Модели конфигурируемы (сильная + дешёвая).
- **Фоновая генерация:** asyncio-таски в процессе FastAPI (без Celery/Redis на
  этом масштабе). Устойчивость к рестартам — через статус в БД (`generated_count`).

## 4. Архитектура

```
Пользователь → Next.js → FastAPI → ├─ Postgres (users, sessions, questions, answers, профили)
                                    └─ OpenAI (генерация + self-проверка + финальная рекомендация)
```

Три подсистемы на бэкенде:

1. **Auth** — регистрация (логин + пароль + код доступа), вход, сессии, дневной
   rate-limit.
2. **Generation** — план тем под уровень/режим, генерация пула батчами,
   self-проверка, стриминг готовых вопросов.
3. **Assessment** — приём ответов, детерминированная оценка, подсчёт %, обновление
   профиля по темам, финальная LLM-рекомендация.

## 6. Поток генерации, self-проверка и стриминг

**Шаг 0. Старт сессии.** Пользователь выбирает уровень и режим. Бэкенд:
проверяет дневной лимит; строит план тем (для `exam` — официальные пропорции,
для `adaptive` — слабые темы из `topic_competency`); создаёт `test_sessions`
со `status=generating`.

**Шаг 1. Генерация (батчи по 10), сильная модель.** Один LLM-вызов на батч,
structured output (JSON-mode). Системный промпт (требования к качеству, формат,
правила дистракторов) — фиксированный, **кэшируется** (prompt caching).
Пользовательский промпт — темы и количество в батче. Каждый вопрос приходит сразу
с `explanation`.

**Шаг 2. Self-проверка, дешёвая модель.** Каждый вопрос → лёгкий вызов:
корректность, однозначность, верность `correct_keys`. `passed` → пишется в
`questions`, `generated_count++`. `rejected` → отбрасывается, слот в очередь на
перегенерацию (макс. 2 повторные попытки; после — берём лучшую попытку и логируем).

**Шаг 3. Стриминг.** Фронт получает прогресс и готовые вопросы (SSE или поллинг
`GET /sessions/{id}/status`). Пользователь видит «45 / 120», может **отвечать на
готовые вопросы сразу** (ответы сохраняются). Долистал до неготового → UI
блокирует переход, «вопрос генерируется…».

**Шаг 4. Готовность пула.** `generated_count == total_questions` → `status=ready`,
проставляется `timer_started_at` — **официальный таймер стартует здесь**. Ответы,
данные до этого момента, остаются засчитанными.

**Обработка ошибок:** ошибка батча (timeout/невалидный JSON) → ретрай с
экспоненциальной задержкой (до 3 попыток). Неустранимая ошибка → `status=failed`,
пользователю «не удалось подготовить тест», сессия **не** засчитывается в дневной
лимит. Генерация — фоновая asyncio-таска. Закрыл вкладку и вернулся → статус и
готовые вопросы из БД, генерация продолжается.

## 7. Оценка стоимости (токены)

Инженерная оценка по допущениям (длина вопроса, ~35% вопросов с артефактом,
~15% перегенерации). **Порядок величин — сверить с актуальными ценами OpenAI при
реализации (cutoff January 2026).**

**Один вопрос (генерация):** ~500 output-токенов (вопрос ~80, варианты ~120,
артефакт в среднем ~90, эталон+мета ~30, объяснение ~150, обвязка ~30).
**Self-проверка одного вопроса:** ~750 input + ~120 output.

**Сессия «Специалист» (120 вопросов):**

| Этап | Input | Output |
|---|---|---|
| Генерация (12 батчей) | ~21 600 | ~60 000 |
| Self-проверка (120) | ~90 000 | ~14 400 |
| Перегенерация (~15%) | ~9 500 | ~2 500 |
| Финальная рекомендация | ~2 000 | ~600 |
| **ИТОГО** | **~123 000** | **~77 500** |

≈ **200 000 токенов на сессию «Специалист»**; «Базовый» (80) ≈ **135 000**.

Грубая стоимость сессии «Специалист»: ~$1.1 (флагман) … ~$0.07 (mini-класс).
**Prompt caching** срезает значительную часть input (системный промпт общий для
батчей). Разделение моделей (сильная — генерация, дешёвая — проверка) снижает
стоимость в разы.

## 8. UI и экраны (Next.js), стиль A — деловой строгий (IBS)

Визуал: бело-синяя гамма, сетка вопросов слева, центральная карточка, видимая
блокировка несгенерированных вопросов.

1. **Регистрация** — логин, пароль, код доступа (при неверном коде — отказ).
2. **Вход** — логин + пароль.
3. **Дашборд** — выбор уровня и режима, кнопка «Начать»; история сессий (дата,
   уровень, %, сдал/не сдал); профиль по темам (accuracy-бары). При исчерпании
   дневного лимита — старт заблокирован с пояснением.
4. **Подготовка теста** — прогресс генерации («45 / 120»); как только есть первые
   вопросы — кнопка «Начать отвечать».
5. **Прохождение теста:**
   - текст вопроса + артефакт (`<pre>` для кода/JSON/SQL/XML, Mermaid для диаграмм);
   - варианты: radio (single) / checkbox (multi);
   - навигация (сайдбар-сетка с отметками «отвечено», кнопки назад/вперёд);
   - таймер (стартует при готовности пула; до этого — индикатор «идёт подготовка,
     таймер ещё не запущен»);
   - блок при попытке открыть неготовый вопрос;
   - «Завершить тест» + авто-завершение при истечении таймера.
6. **Результаты** — итоговый % крупно, сдал/не сдал по порогу, разбивка по темам
   (accuracy-бары), разбор каждого вопроса (ваш/правильный ответ, подсветка, +
   готовое объяснение), блок «Персональная рекомендация» (финальный LLM-текст).

## 9. Явно вне области (YAGNI / на будущее)

- Банк проверенных вопросов с переиспользованием (эволюция self-проверки) —
  заложено идейно, но не в первой версии.
- Celery/Redis-воркер — добавить только если масштаб вырастет.
- Открытый ввод (свободный текст) — исключён, ломает детерминированность оценки.
- Сгенерированные картинки-диаграммы — исключены в пользу Mermaid.
- Email/SMTP, magic-link, SSO — не нужны (логин + пароль).

## 10. Открытые параметры конфигурации (значения — при реализации)

- Конкретные модели OpenAI (сильная / дешёвая).
- Значение кода доступа.
- Дневной лимит сессий на пользователя.
- Размер батча генерации (по умолчанию 10).
- Объём адаптивного теста и порог «слабой темы».
- Официальная матрица пропорций тем по уровням (из исследовательского файла).
