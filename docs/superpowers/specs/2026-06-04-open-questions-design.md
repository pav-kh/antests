# Открытые вопросы с LLM-оценкой — Дизайн

**Дата:** 2026-06-04
**Статус:** утверждён к реализации
**Контекст:** дополнение к тренажёру сертификации (см. 2026-06-03-certification-trainer-design.md)

## 1. Назначение

Добавить ровно **2 открытых вопроса** (свободный текст) в каждый тест, сверх
официальных закрытых. LLM оценивает открытые ответы и даёт развёрнутую обратную
связь. Обратная связь **не влияет** на проходной балл — он по-прежнему считается
детерминированно только по закрытым вопросам.

Это сознательное изменение раннего решения исключить открытый ввод: тогда его не
брали, чтобы не вводить недетерминированную оценку. Здесь недетерминированность
изолирована в отдельной секции, которая не затрагивает «сдал/не сдал».

## 2. Ключевые продуктовые решения

1. **Вариант B (оценка отдельно):** открытые дают LLM-обратную связь, но НЕ входят
   в проходной %. Закрытые определяют «сдал/не сдал» детерминированно (как сейчас).
2. **Ровно 2 открытых** на тест.
3. **Сверх закрытых:** 80+2 (Базовый), 120+2 (Специалист). Официальная закрытая
   часть и её пропорции тем не меняются.
4. **Оценка при «Завершить»**, вместе с подсчётом результата: один лоадер,
   LLM-вызовы (2 оценки открытых + финальная рекомендация) распараллелены.
5. **Хранение в существующих таблицах:** `type="open"` в `questions`, ответ в
   `answers`. Без новых таблиц.
6. **Пустой ответ → заглушка без LLM:** если `answer_text` пустой / < 10
   непробельных символов — фидбэк-заглушка «Вы не ответили…» + что стоило
   раскрыть (из rubric), без вызова судьи.
7. **Секретность:** `rubric` (критерии судьи) не уходит на фронт, как `correct_keys`.

## 3. Модель данных (изменения)

### `questions`
| Поле | Изменение |
|---|---|
| `type` | допускается значение `open` (плюс к `single`/`multi`). Хранится как String — менять схему enum не нужно, только код-валидацию. |
| `rubric` | **новое** `text NULL` — критерии оценки / эталон для LLM-судьи. Только у открытых. НЕ отдаётся в `/questions`. |
| `options`, `correct_keys` | у открытых = `[]`. |
| `stem` | у открытого = текст задания (кейс). |
| `explanation` | у открытого = разбор «что раскрывает хороший ответ» (показывается на результатах). |

### `answers`
| Поле | Изменение |
|---|---|
| `answer_text` | **новое** `text NULL` — текст ответа пользователя (для открытых). |
| `feedback` | **новое** `text NULL` — обратная связь LLM-судьи. |
| `selected_keys` | у открытых = `[]`. |
| `is_correct` | у открытых = **`null`** → исключает из детерминированного подсчёта. |

**Миграция (Alembic):** `rubric` в `questions`; `answer_text`, `feedback` в `answers`.

### Критично: подсчёт только по закрытым
`score_percent` и `total` для проходного балла считаются **только** по
`type in (single, multi)`. Сейчас `total = len(questions)` — изменить на число
закрытых. `topic_breakdown` — тоже только по закрытым. Иначе 2 открытых исказят %.

## 4. Генерация (Generation)

- После заполнения закрытого пула — **отдельный вызов** на 2 открытых:
  LLM возвращает `{stem, rubric, explanation}` (без options/correct_keys).
- Промпт: открытый вопрос-кейс для системного аналитика (пример: «Клиенты часто
  пишут повторно… какие вопросы задать клиенту и какие решения предложить?»);
  `rubric` = что должен раскрыть хороший ответ.
- Открытые НЕ участвуют в shuffle закрытых — идут **в конце** (seq после всех
  закрытых), отдельной секцией.
- Текущий self-валидатор (заточен под «верный ответ / дистракторы») к открытым
  НЕ применяется; для них — лёгкая проверка (stem осмыслен, rubric непустой).
- Открытые НЕ участвуют в artifact-квоте и dedup закрытых (отдельный путь).

## 5. Оценка (Assessment)

- `submit_answer` для открытого: пишет `answer_text` (не `selected_keys`),
  `is_correct=null`. Резолвинг открытого/закрытого по `question.type`.
- `finish_session`:
  1. Закрытые — детерминированный подсчёт (как сейчас); `score_percent`/`passed`
     ТОЛЬКО по ним.
  2. Для каждого открытого: пустой/короткий `answer_text` → заглушка без LLM;
     иначе → судья (`stem` + `rubric` + текст → `feedback`).
  3. Оценки открытых + финальная рекомендация — **параллельно** (`asyncio.gather`).
  4. `feedback` → `answers.feedback`.
- **Новый метод клиента:** `OpenAIClient.judge_open(stem, rubric, answer) -> str`
  (free text). Дешёвая модель (`validate_model`).

### `get_results`
- Добавляется секция `open_questions: [{stem, answer_text, feedback, explanation}]`
  — отдельно от `questions` (закрытых).
- `score_percent`/`topic_breakdown` — без открытых.

## 6. Frontend

### Экран теста (`test/[id]/page.tsx`, `QuestionCard`)
- Открытые в навигации идут последними, помечены иначе (видно «открытый»).
- `QuestionCard` для `type="open"`: `<textarea>` вместо radio/checkbox; ответ
  сохраняется через тот же `submitAnswer` (поле `answer_text`).
- Таймер и блокировка несгенерированных — как есть.

### Экран результатов (`results/page.tsx`)
- Новая секция **«Открытые вопросы»** после разбора закрытых: задание, ваш ответ,
  обратная связь (feedback), разбор (explanation). Без «верно/неверно».
- Балл/по темам/рекомендация/разбор закрытых — без изменений.

### API-клиент
- `submitAnswer` принимает `selected_keys` ИЛИ `answer_text`.
- Тип `Results` получает поле `open_questions`.

## 7. Тестирование

- **Backend:** open валиден в схеме; `finish` считает % только по закрытым;
  пустой ответ → заглушка без LLM; непустой → судья (fake client); `rubric` не в
  `/questions`; `feedback` в `/results`; resume восстанавливает `answer_text`.
- **Frontend:** `QuestionCard` рендерит `<textarea>` для open; results показывает
  секцию открытых.
- **Live:** реальный тест → 2 открытых появились → ответить → осмысленный
  feedback → балл не изменился от открытых.

## 8. Вне области (YAGNI)
- Балльная оценка открытых / влияние на проходной балл — исключено (решение B).
- Отдельные таблицы для открытых — не нужны.
- Стриминг оценки открытых — не нужен (показываем при finish, один лоадер).
- Редактирование/дозапрос фидбэка — нет.

## 9. Открытые параметры (значения при реализации)
- Порог «пустого» ответа (по умолчанию < 10 непробельных символов).
- Модель для `judge_open` (по умолчанию `validate_model`).
- Точная формулировка промпта генерации открытых и промпта судьи.
