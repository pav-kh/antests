# 70% multi на base/specialist + блочное форматирование открытых вопросов — Design

**Дата:** 2026-06-19
**Статус:** Одобрено пользователем
**Контекст:** Два запроса по тестам base/specialist: (1) 70% закрытых вопросов должны быть multi (несколько верных вариантов), как уже сделано для ba; (2) открытые вопросы на странице результатов (и на экзамене) рендерятся одной сплошной жирной строкой — нужно читаемое блочное форматирование.

---

## Цель

base/specialist получают ту же мягкую ~70%-multi квоту, что и ba. Открытые вопросы рендерятся структурированными блоками (шапка-мета приглушена, кейс обычным текстом, «Задание / Фокус ответа / Критерии оценки» — отдельными блоками с метками) на экзамене И на странице результатов.

## Решения (зафиксированы с пользователем)

1. **Multi 70%** для base и specialist (та же мягкая квота, что у ba).
2. **Блочное форматирование** открытых вопросов (разбить сплошной stem на секции).
3. **Парсинг на фронте** (без изменений бэкенда/БД, работает со старыми сессиями).
4. **Применить в двух местах**: экзамен (`QuestionCard`) и результаты (`results/page`).
5. **Шапка** показывается приглушённой строкой (тема + лимит «до 2500…» мелким muted-текстом); кейс и три блока — основной контент.

## Часть 1 — Multi-квота (бэкенд)

`backend/app/generation/planner.py`:
```python
LEVEL_MULTI_TARGET = {"ba": 0.7, "base": 0.7, "specialist": 0.7}
```
Генератор читает `LEVEL_MULTI_TARGET.get(session.level)` и пробрасывает `multi_ratio` в `generate_batch`, который добавляет в промпт инструкцию делать multi.

**Совместимость:** ba уже на 0.7 (без изменений). base/specialist раньше не имели стиринга → теперь получают. Старые сессии не затрагиваются.

### Часть 1b — ЖЁСТКАЯ квота (добавлено после живого smoke)

Живой smoke показал, что мягкой квоты недостаточно: LLM недодаёт multi (base 34%, specialist 52% вместо ~70%). Решение пользователя: **усилить промпт + добавить жёсткую квоту с догенерацией**.

**Усиление промпта** (`openai_client.py`, multi-блок в `generate_batch`): формулировка делается более настойчивой — «БОЛЬШИНСТВО вопросов (≥{pct}%) ДОЛЖНЫ быть multi», с явным примером (2–3 верных из 4–5) и запретом делать почти все single. Сама по себе не гарантирует, но поднимает базовую долю.

**Жёсткая догенерация** (`generator.py`, новый шаг ПОСЛЕ per-topic закрытого цикла и ДО shuffle):
- Параметры (константы в generator.py): `MULTI_TOPUP_ROUNDS = 3` (макс. раундов добора).
- После генерации всех закрытых: посчитать `multi_count` и `target = ceil(multi_ratio * total_closed)` (только если `multi_ratio` задан).
- Пока `multi_count < target` и раундов осталось: взять одну хранимую `single`-строку (любого топика), перегенерировать её как multi-вопрос на ТОМ ЖЕ topic_id (вызов `generate_batch(level, mode, [(topic, 1)], multi_ratio=1.0)` — попросить именно multi), и если вернулся валидный multi-вопрос (не дубликат, ≥2 correct_keys) — ЗАМЕНИТЬ содержимое существующей строки (stem/options/correct_keys/explanation/type="multi"), сохранив seq. Инкремент `multi_count`.
- За раунд можно конвертировать несколько single (итерируем по single-строкам, пока не добьём target или не кончатся single в этом раунде); раунд = один проход. Бюджет — `MULTI_TOPUP_ROUNDS` проходов.
- **Фоллбэк:** после исчерпания раундов — принять достигнутую долю (НЕ зацикливаться, НЕ падать). Сессия всё равно `ready`.
- Конвертация заменяет single на multi → общее число закрытых и покрытие тем не меняются.
- Дедуп: новый multi-stem проверяется против `seen_stems`; дубликат — пропустить, попробовать следующий single/раунд.

**Совместимость:** только base/specialist/ba (у кого есть `multi_ratio`). Уровни без `multi_ratio` — шаг пропускается (как раньше). Старые сессии не затрагиваются (логика только при генерации). Шаг идёт ДО shuffle, поэтому seq-перестановка и open-вопросы не затрагиваются.

## Часть 2 — Блочное форматирование (фронтенд)

### Текущая структура stem (из `build_open_stem`)
```
Ответ: до 2500 знаков с пробелами; достаточно тезисного, структурированного ответа.
Тип: открытый кейс. {topic_title}

{case}

Задание: {task}
Фокус ответа: {focus}
Критерии оценки: {criteria_visible}
```

### Компонент 1: `frontend/src/lib/openStem.ts` — чистый парсер

```typescript
export interface ParsedOpenStem {
  topicTitle: string;   // из "Тип: открытый кейс. {topic_title}"
  answerHint: string;   // из "Ответ: ..." (лимит/формат)
  case: string;         // текст кейса (между шапкой и "Задание:")
  task: string;         // из "Задание: ..."
  focus: string;        // из "Фокус ответа: ..."
  criteria: string;     // из "Критерии оценки: ..."
}

export function parseOpenStem(stem: string): ParsedOpenStem | null
```
- Возвращает `null`, если stem не содержит ожидаемых меток (старый/нестандартный формат) → вызывающий код делает fallback.
- Парсит по якорям: строка `Ответ:` → answerHint; `Тип: открытый кейс.` → topicTitle (часть после точки); блок между шапкой и `Задание:` → case; `Задание:` → task; `Фокус ответа:` → focus; `Критерии оценки:` → criteria.
- Каждая метка извлекается от ПЕРВОГО её вхождения до начала следующей метки. Метки идут в фиксированном порядке (Ответ → Тип → кейс → Задание → Фокус ответа → Критерии оценки); берётся первое вхождение каждого якоря, чтобы случайное совпадение слова в тексте кейса не сбило разбор. Чистая функция, без React — тестируется юнитами.

### Компонент 2: `frontend/src/components/OpenStem.tsx` — рендер

```tsx
export function OpenStem({ stem }: { stem: string })
```
- Вызывает `parseOpenStem(stem)`.
- **Если распарсилось** — рендерит блоки:
  - Шапка: `answerHint` приглушённой мелкой строкой (стиль muted, ~12px, `var(--muted)`); тема `topicTitle` — заметным подзаголовком.
  - Кейс: обычный читаемый абзац (не жирный, нормальный размер), `white-space: pre-line`.
  - Три блока (Задание / Фокус ответа / Критерии оценки): каждый — жирная метка-заголовок + обычный текст под ней; вертикальные отступы между блоками.
- **Если `null`** (fallback) — рендерит весь `stem` как сейчас: обычный текст с `white-space: pre-line` (ничего не теряется).

### Применение
- **Экзамен** `frontend/src/components/QuestionCard.tsx`: для `question.type === "open"` заменить текущий `<h3 ... whiteSpace pre-line>{question.stem}</h3>` на `<OpenStem stem={question.stem} />`. Закрытые вопросы — без изменений (остаются с `<h3>`).
- **Результаты** `frontend/src/app/test/[id]/results/page.tsx`: заменить `<h4 style={{ margin: "8px 0" }}>{o.stem}</h4>` (строка ~115) на `<OpenStem stem={o.stem} />`. (Это место сейчас хуже всего — нет даже pre-line.)

## Совместимость / без миграций

- Бэкенд: только одна строка `LEVEL_MULTI_TARGET`. Никаких миграций, схем, изменений API.
- Парсинг работает с любым stem, собранным `build_open_stem` (текущим и будущим). Старые сессии рендерятся корректно (или через fallback, если формат отличается).
- Закрытые вопросы, скоринг, секции результатов — не затрагиваются.

## Тестирование

- `parseOpenStem` (юнит): на полном stem из `build_open_stem` извлекает все 6 полей корректно (topicTitle="Системное мышление", answerHint содержит «2500», case/task/focus/criteria — соответствующий текст); на строке без меток возвращает `null`.
- `OpenStem` (RTL): распарсенный stem рендерит метки-заголовки «Задание»/«Фокус ответа»/«Критерии оценки» и тему; непарсящийся stem рендерит исходный текст (fallback), textbox-ничего не ломается. (Открытый вопрос на экзамене содержит textarea — `OpenStem` только заменяет рендер stem, textarea остаётся.)
- multi: `LEVEL_MULTI_TARGET` содержит base/specialist=0.7; существующий `test_multi_quota.py` — добавить/обновить кейс, что для base промпт получает multi-инструкцию (раньше тест проверял, что base её НЕ получает — этот ассерт надо инвертировать).
- Живой smoke: base/specialist — доля multi среди закрытых заметно выше 50% (ориентир ~70%); открытый вопрос на экзамене и на результатах виден блоками (тема, кейс, Задание/Фокус/Критерии раздельно).

## Вне scope (YAGNI)

- Изменение бэкенд-структуры stem / хранение секций отдельными полями (решено: парсинг на фронте).
- Markdown-рендеринг внутри секций (текст плоский).
- Изменение долей multi для ba (уже 0.7).
- Жёсткая multi-квота с отбрасыванием (мягкой достаточно).
- Форматирование закрытых вопросов (только открытые).

## Замечание по существующему тесту multi-квоты

`backend/tests/test_multi_quota.py` сейчас содержит `test_generate_batch_no_multi_instruction_when_ratio_none`, который вызывает `generate_batch("base", ...)` без `multi_ratio` и проверяет ОТСУТСТВИЕ multi-инструкции. Это по-прежнему валидно как unit (проверяет поведение `generate_batch` при `multi_ratio=None`) — он тестирует метод, а не то, какой ratio получает base от генератора. Также есть `test_ba_has_multi_target`, который ассертит `"base" not in LEVEL_MULTI_TARGET` — этот ассерт станет ложным и его НУЖНО обновить (base/specialist теперь в словаре с 0.7).
