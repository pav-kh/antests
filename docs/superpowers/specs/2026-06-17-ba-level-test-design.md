# BA-уровень: тест по бизнес-анализу (42 вопроса) — Design

**Дата:** 2026-06-17
**Статус:** Одобрено пользователем
**Контекст:** Тренажёр сейчас имеет уровни `base` (80 вопросов) и `specialist` (120). Нужен третий уровень — тест строго по бизнес-анализу: 42 вопроса (40 закрытых + 2 открытых), максимальный охват BA-тем, преимущественно вопросы с несколькими верными вариантами (multi). Цель: после тренировки пользователь уверенно проходит официальные BA-сертификации и собеседования.

---

## Цель

Новый уровень `ba` — отдельный тест по бизнес-анализу, покрывающий все 6 областей знаний BABOK + Agile-BA + софт-скиллы, чтобы закрыть пробелы текущего (систем-аналитического) набора тем. 42 вопроса: 40 закрытых (преимущественно multi) + 2 открытых кейса (механизм пула из предыдущей фичи).

## Решения (зафиксированы с пользователем)

1. **Новый уровень `ba`** — рядом с base/specialist, выбирается на дашборде третьей кнопкой.
2. **42 вопроса** = 40 закрытых + 2 открытых. `LEVEL_TOTALS["ba"]=40` (закрытые); открытые добавляются механизмом пула (+2 сверх закрытых), `total_questions` показывает закрытые = 40 (как в base/specialist).
   - **Замечание по UI (40 vs 42):** как и в base/specialist, навигация по факту покрывает все вопросы через `maxSeq` (из прошлой фичи), поэтому пользователь физически видит и проходит все 42 (закрытые 1..40 + открытые 41,42). Счётчик «Вопросы · N» в сайдбаре показывает `total_questions`=40 (оцениваемая закрытая часть), а открытые 2 — бонусная секция сверх, как уже устроено для base/specialist. Это сознательно консистентно с существующими уровнями; отдельной правки «показать 42» не делаем.
3. **18 тем** = 10 существующих + 8 новых BA-тем. Веса для `ba` через ключ `"ba"` в `proportions` каждой темы (единый список TOPICS).
4. **~70% multi** среди закрытых вопросов уровня `ba`. Реализация — мягкая квота (как artifact-квота): цель в промпте + учёт по факту, без жёсткого отбрасывания.
5. **Открытые** — без изменений (пул seed+LLM, 2 сэмплятся). Seed-кейсы (БА-1, БА-2) тематически подходят BA-тесту.
6. **RACI/матрицы разведены по углу:** `stakeholders` — RACI как инструмент анализа стейкхолдеров (власть/интерес, вовлечённость); `methodology` — RACI как распределение ролей по задачам проекта. Оба угла присутствуют, дублей нет.

## Архитектура

`level` — строка, валидируется в роутере. Уровень управляет: тоталом (`LEVEL_TOTALS`), таймлимитом (`LEVEL_TIME_LIMITS`), планом тем (через `proportions[level]`), долей multi (новый `LEVEL_MULTI_TARGET`), компетенциями (по topic_id, уже generic). Никаких миграций БД: новый level — значение строки, новые темы — Python-константы.

### Поток (создание ba-сессии)

```
create_session(level="ba", mode="exam"):
    plan = plan_exam("ba")              # веса = {t.id: t.proportions["ba"]}; темы с весом 0 отсеиваются
    total = LEVEL_TOTALS["ba"]          # 40
    session.time_limit_sec = LEVEL_TIME_LIMITS["ba"]   # 90*60
Generator.run(plan):
    # закрытые: как сейчас, но generate_batch получает want_multi_ratio из LEVEL_MULTI_TARGET["ba"]=0.7
    # открытые: пул seed+LLM, сэмплим 2 (без изменений)
```

`mode`: для `ba` используем `exam` (фиксированный план по весам тем — максимальный охват). `adaptive` для `ba` тоже допустим технически (план по слабым темам), но кнопка на дашборде создаёт `ba`+`exam`. Adaptive-ba не запрещаем, просто UI ведёт в exam.

## Компоненты

### 1. `backend/app/generation/topics.py` — +8 тем, +ключ "ba"

К каждой из 10 существующих тем добавить ключ `"ba"` в `proportions` (вес из таблицы ниже). Добавить 8 новых `Topic` с `proportions={"base":0.0, "specialist":0.0, "ba":<вес>}`.

Новые темы (id — title — subtopics):

```
stakeholders — Анализ и управление стейкхолдерами
  - Идентификация стейкхолдеров, реестр (stakeholder list)
  - Матрица власть/интерес, карты вовлечённости
  - RACI как инструмент анализа стейкхолдеров (кто вовлечён, влияние/интерес)
  - Персоны и анализ потребностей
  - План коммуникаций и управление ожиданиями

strategy — Стратегический анализ и бизнес-обоснование
  - Бизнес-потребность и определение проблемы (problem statement)
  - Анализ текущего/целевого состояния (current/future state, gap analysis)
  - SWOT, бизнес-модель, цели и метрики
  - Бизнес-кейс, оценка выгод, ROI/NPV/TCO
  - Оценка и управление рисками, стратегия изменений

process_analysis — Анализ и улучшение бизнес-процессов
  - As-Is / To-Be анализ процессов
  - Выявление узких мест, потерь, корневых причин (5 почему, Ишикава)
  - Реинжиниринг и оптимизация процессов
  - Метрики процессов (время цикла, стоимость, KPI)
  - Связь процессов с требованиями и решением

elicitation — Выявление требований: техники
  - Интервью и анкетирование (surveys)
  - Воркшопы и фасилитация, мозговой штурм
  - Наблюдение, анализ документов, анализ интерфейсов
  - Прототипирование как техника выявления
  - Подтверждение и согласование результатов

solution_value — Оценка и приёмка решения
  - KPI и метрики ценности решения
  - Измерение производительности решения, benefits realization
  - Анализ ограничений решения и организации
  - Приёмочное тестирование (UAT), критерии приёмки
  - Рекомендации: расширять / заменить / вывести из эксплуатации

agile_ba — Бизнес-анализ в Agile
  - User Stories, формат «Как… я хочу… чтобы…», INVEST
  - Критерии приёмки (Acceptance Criteria), Definition of Done
  - Управление product backlog, epics, груминг
  - Приоритизация: MoSCoW, Kano, RICE, value-vs-effort, WSJF
  - Story mapping, оценка (story points, planning poker)

ba_planning — Планирование и мониторинг БА-работ
  - Выбор подхода к бизнес-анализу (predictive/adaptive)
  - Планирование вовлечения стейкхолдеров и governance
  - Управление информацией о требованиях
  - Метрики и улучшение эффективности БА-работ

soft_skills — Коммуникации и софт-скиллы аналитика
  - Фасилитация и проведение встреч
  - Деловая коммуникация и переговоры
  - Презентация и защита решений
  - Работа с конфликтами и заинтересованными сторонами
  - Аналитическое и системное мышление
```

Существующая тема `methodology`: подтему «RACI» переформулировать (только концептуально — это контекст для промпта) на «RACI как распределение ролей по задачам проекта», чтобы развести с углом `stakeholders`. На уровне кода subtopics — это строки; правка одной строки.

### 2. `backend/app/generation/planner.py` — `LEVEL_TOTALS["ba"]=40`

Добавить `"ba": 40`. `plan_exam`/`plan_adaptive` уже generic по level и набору весов — менять логику не нужно. Проверка: `plan_exam("ba")` с 18 темами на 40 вопросов — каждая тема ≥1 (logic «missing topics» добивает редкие темы за счёт самой крупной).

### 3. `backend/app/generation/service.py` — таймлимит + проброс multi-target

- `LEVEL_TIME_LIMITS["ba"] = 90 * 60` (90 минут — вдвое меньше вопросов, чем base/120мин).
- При генерации передать желаемую долю multi в Generator (см. ниже). `create_session` уже возвращает `(session, plan)`; level известен, доля берётся из `LEVEL_MULTI_TARGET`.

### 4. Доля multi — мягкая квота

- Новая константа (в planner.py или generator.py): `LEVEL_MULTI_TARGET = {"ba": 0.7}`. Для base/specialist ключа нет → поведение НЕ меняется (как сейчас, без квоты).
- `Generator.run` читает `LEVEL_MULTI_TARGET.get(session.level)` (может быть None). Если задано — пробрасывает в `generate_batch(..., multi_ratio=...)`.
- `generate_batch` / `build_generation_system_prompt`: когда `multi_ratio` задан, добавить в промпт инструкцию: «Не менее ~70% вопросов в этом батче делай типа multi (несколько верных вариантов, обычно 2–3 из 4–5); остальные — single». Когда не задан — текущий промпт без изменений.
- **Мягкий учёт, без отбрасывания:** не валим вопрос, если single там, где хотели multi. Доля — ориентир, как artifact-квота (target, не hard cap). Это сохраняет наполнение тем и согласуется с существующим паттерном.

### 5. Frontend

- `frontend/src/lib/types.ts`: `Level = "base" | "specialist" | "ba"`.
- `frontend/src/app/dashboard/page.tsx`: третья кнопка уровня — «Бизнес-анализ». Маппинг подписи.
- `frontend/src/app/test/[id]/page.tsx`: подпись уровня — добавить ветку `ba → "Бизнес-анализ"` (сейчас тернарник base/specialist).
- Никаких изменений в логике экзамена/результатов — open-pool и closed-only скоринг переиспользуются.

## Обработка ошибок / совместимость

- **Миграций нет:** level — строка; новые темы — Python-константы; компетенции generic по topic_id.
- **base/specialist не затронуты:** новые темы имеют там вес 0; multi-target для них не задан.
- **Открытые вопросы:** механизм пула без изменений; seed-кейсы подходят BA-тесту.
- **Старые сессии:** не затрагиваются (новый level только для новых сессий).
- **plan_exam("ba") на 40 вопросов и 18 тем:** 40 ≥ 18, поэтому каждая тема получает ≥1; редкие темы (security 0.01) добиваются логикой «missing» за счёт самой крупной.

## Тестирование

- `topics`: 18 тем; новые 8 имеют base=specialist=0 и ba>0; старые 10 имеют ba>0; RACI-подтемы разведены (stakeholders vs methodology) — проверить наличие маркеров.
- `planner`: `plan_exam("ba")` суммируется в 40; каждая из 18 тем присутствует (≥1); base/specialist планы НЕ изменились (regression — те же суммы и темы, что были).
- `LEVEL_TOTALS["ba"]==40`, `LEVEL_TIME_LIMITS["ba"]==5400`.
- `create_session(level="ba", mode="exam")`: total=40, time_limit=5400, plan покрывает 18 тем.
- multi-target: `build_generation_system_prompt`/`generate_batch` при `multi_ratio=0.7` включают инструкцию о multi; без него — старый промпт. (Юнит на формирование промпта; долю по факту не ассертим жёстко — мягкая квота.)
- router: `level="ba"` валиден; невалидный level по-прежнему 400.
- Frontend: дашборд показывает 3 кнопки уровня; test-page подпись «Бизнес-анализ» для ba.
- Live smoke: создать ba-сессию, дождаться ready, проверить: 40 закрытых + 2 открытых = 42; темы покрыты широко (много разных topic_id, включая новые BA-темы); доля multi заметно выше 50% (ориентир ~70%, не жёстко); открытые без rubric.

## Вне scope (YAGNI)

- Adaptive-режим как отдельная кнопка для ba (технически работает, но UI ведёт в exam).
- Жёсткая квота multi с пост-отбрасыванием (мягкой достаточно).
- Перебалансировка весов base/specialist.
- Отдельные seed-кейсы только для ba (текущие 2 подходят).
- Изменение таймлимитов base/specialist.
