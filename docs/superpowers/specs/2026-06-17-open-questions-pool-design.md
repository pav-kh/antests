# Пул открытых вопросов (fixed + LLM) с видимыми критериями — Design

**Дата:** 2026-06-17
**Статус:** Одобрено пользователем
**Контекст:** Расширение фичи открытых вопросов (см. `2026-06-04-open-questions-design.md`). Пользователь нашёл 2 реальных открытых кейса из настоящей сертификации БА (базовый уровень) и хочет, чтобы тренажёр (1) генерировал вопросы в точно таком же формате и (2) включал эти 2 реальных кейса.

---

## Цель

Открытые вопросы тренажёра должны соответствовать формату реальной сертификации БА: структура «кейс → задание → фокус ответа → критерии оценки», лимит до 2500 знаков с пробелами, видимые пользователю критерии. В пул кандидатов входят реальные образцы кейсов и LLM-сгенерированные вопросы; на тест берутся 2 случайных.

## Решения (зафиксированы с пользователем)

1. **Сочетание fixed + LLM:** «Fixed + одна LLM-генерация». При генерации сессии формируется пул кандидатов: fixed-кейсы (реальные) + несколько LLM-сгенерированных. Из пула берутся 2 случайных.
2. **Видимость критериев:** «Показывать как в оригинале». «Фокус ответа» и «Критерии оценки» — часть видимой формулировки вопроса (как на скринах реального теста).
3. **Скрытый rubric:** «Видимые + скрытый rubric». Пользователь видит критерии в `stem`; судья LLM дополнительно получает более подробный скрытый `rubric`.
4. **Хранение видимых полей:** «Вшить в stem (без миграции)». «Фокус» и «Критерии» собираются в единый текст `stem`. Новых колонок БД нет, миграция не нужна.

## Архитектура

Открытые вопросы по-прежнему: `type="open"`, `topic_id="open"`, seq после закрытого пула, не влияют на pass/fail (см. предыдущий спек). Меняется только то, ОТКУДА берутся 2 открытых вопроса и КАК выглядит их `stem`.

### Поток данных при генерации сессии

```
generator.run():
  … (закрытые вопросы как сейчас: генерация, валидация, shuffle) …
  pool = []
  pool += SEED_OPEN_QUESTIONS                      # fixed-кейсы (Python-константа)
  try:
      pool += await client.generate_open_questions(level, count=LLM_OPEN_CANDIDATES)  # LLM-кандидаты
  except Exception:
      logger.exception(...)                        # пул не пустеет — fixed хватает
  rng = random.Random(str(session.id))             # тот же детерминизм, что у закрытого shuffle
  chosen = sample_without_replacement(pool, 2, rng)
  for oq in chosen:
      seq += 1
      db.add(Question(seq=seq, topic_id="open", type="open",
                      stem=oq.stem, rubric=oq.rubric, explanation=oq.explanation, …))
  session.generated_count = seq                    # разблокирует open в UI (как сейчас)
```

## Компоненты

### 1. `backend/app/generation/open_seed.py` (новый)

```python
SEED_OPEN_QUESTIONS: list[OpenQuestion]
```

Python-константа со списком готовых `OpenQuestion`. На старте — 2 реальных кейса (БА-1, БА-2). Каждый seed-кейс хранит:
- `stem` — уже собранный форматированный текст (кейс + задание + фокус + критерии), как видит пользователь.
- `rubric` — подробный скрытый критерий для судьи.
- `explanation` — разбор, что отличает сильный ответ (показывается на результатах).

Не таблица БД, не Alembic-миграция — обычный модуль. Будущие кейсы добавляются дописыванием в список.

**Тексты двух реальных кейсов** (очищены от OCR-артефактов; собранный `stem`):

**БА-1 — «От бизнес-проблемы к требованиям»:**
```
Ответ: до 2500 знаков с пробелами; достаточно тезисного, структурированного ответа.
Тип: открытый кейс. От бизнес-проблемы к требованиям.

В финтех-компании растёт количество обращений клиентов по статусу заявки на продукт. Руководитель клиентского сервиса говорит: «Клиенты часто пишут повторно, потому что не понимают, где находится заявка и почему она задержалась». Сейчас статусы ведутся в нескольких системах, а операторы вручную уточняют информацию у смежных подразделений.

Задание: Сформулируйте до 5 уточняющих вопросов к заинтересованным сторонам, до 4 требований верхнего уровня к будущему решению и укажите, какие 1–2 требования должны быть приоритетными для первой реализации и почему.

Фокус ответа: Не нужно проектировать архитектуру. Нужно показать переход от проблемы к проверяемым требованиям: цель, пользователи, статусы, данные, критерии приёмки и приоритетность объёма.

Критерии оценки: понимание бизнес-потребности; качество уточняющих вопросов; проверяемость требований; учёт заинтересованных сторон; обоснование приоритетов; структура ответа.
```

**БА-2 — «Изменение, приёмка и готовность результата»:**
```
Ответ: до 2500 знаков с пробелами; достаточно тезисного, структурированного ответа.
Тип: открытый кейс. Изменение, приёмка и готовность результата.

Перед релизом заказчик просит добавить новый признак «повышенный риск» для заявок. Признак должен отображаться оператору, передаваться в систему проверки и учитываться при ручном согласовании. Часть данных приходит из внешнего сервиса, который иногда отвечает с задержкой.

Задание: Опишите, какие вопросы нужно уточнить до принятия изменения в релиз, и какие критерии приёмки нужно зафиксировать.

Фокус ответа: Нужно не описывать полную процедуру изменения, а выделить влияние на требования, данные, интеграции, статусы, приёмку и риски релиза.

Критерии оценки: выявление влияния изменения; качество вопросов; приёмочные критерии; учёт интеграции и задержек; реалистичность объёма ответа.
```

### 2. `backend/app/generation/openai_client.py` — улучшенный промпт

**Размер пула:** `LLM_OPEN_CANDIDATES = 3` (константа в `generator.py`). Пул = 2 seed + 3 LLM = 5 кандидатов, из которых берутся 2. Этого достаточно для вариативности (иногда выпадут оба реальных, иногда смесь, иногда оба LLM) без лишних LLM-вызовов. Значение легко поднять при росте seed-пула.

`generate_open_questions(level, count=LLM_OPEN_CANDIDATES)`:
- Промпт требует формат реального теста: кейс (практическая ситуация) → задание (что сделать, с числовыми рамками вроде «до 5 вопросов») → фокус ответа (что НЕ нужно и что нужно) → видимые критерии оценки.
- LLM возвращает структурные поля отдельно: `case`, `task`, `focus`, `criteria_visible` (видимые), `rubric` (скрытый, подробнее видимых критериев), `explanation`.
- Код собирает `stem` из `case/task/focus/criteria_visible` по тому же шаблону, что у seed-кейсов (единая функция сборки, DRY).
- JSON-схема (strict:true) расширяется новыми полями; все required.
- Лимит 2500 знаков упоминается в промпте как ориентир объёма кейса/ответа.

### 3. `backend/app/generation/generator.py` — сборка пула и сэмплинг

Блок добавления открытых вопросов меняется: вместо «сгенерировать ровно 2» — «собрать пул (seed + LLM), выбрать 2 случайных детерминированным seed». Запись в БД и `generated_count += chosen` — как сейчас.

### 4. `frontend/src/components/QuestionCard.tsx`

- `stem` открытого вопроса рендерится с сохранением переносов строк (`white-space: pre-line`), чтобы блоки «Задание / Фокус / Критерии» читались как на скринах.
- У textarea — подсказка про лимit «до 2500 знаков с пробелами» (placeholder или вспомогательный текст). Жёсткого ограничения ввода не вводим (реальный тест тоже мягкий — «достаточно тезисного»).

## Сборка stem (общий шаблон)

Единая функция `build_open_stem(topic_title, case, task, focus, criteria_visible) -> str` формирует ВЕСЬ видимый текст, включая шапку. Она — единственный источник формата, используется и для LLM-вопросов, и для seed-кейсов:
```
Ответ: до 2500 знаков с пробелами; достаточно тезисного, структурированного ответа.
Тип: открытый кейс. {topic_title}

{case}

Задание: {task}
Фокус ответа: {focus}
Критерии оценки: {criteria_visible}
```

- **LLM-путь:** `generate_open_questions` просит LLM вернуть `topic_title, case, task, focus, criteria_visible, rubric, explanation`; код вызывает `build_open_stem(...)` и кладёт результат в `OpenQuestion.stem`.
- **Seed-путь:** `open_seed.py` вызывает ту же `build_open_stem(...)` с захардкоженными частями двух реальных кейсов — НЕ хранит сырой собранный текст. Так формат seed и LLM гарантированно идентичен, а правка шаблона затрагивает оба пути сразу (DRY).

(Тексты кейсов в разделе компонента 1 приведены в собранном виде для наглядности; в коде они задаются как части и собираются `build_open_stem`.)

## Обработка ошибок

- **LLM упал:** пул = только `SEED_OPEN_QUESTIONS` (≥2). Сессия всё равно получает 2 открытых. Усиливает устойчивость относительно текущего поведения (раньше сбой → 0 открытых).
- **Пул < 2** (теоретически, если seed урезан и LLM упал): берём сколько есть; открытые — бонусная секция, finish не ломается (как в `test_all_open_session_does_not_crash`).
- **Дедуп:** сэмплинг без повторов по индексу пула (один и тот же объект не выбирается дважды).
- **Скрытность rubric:** без изменений — `rubric` не в `/questions` allowlist и не в `/results`.

## Совместимость со старыми сессиями

Миграции нет. Пул строится только при генерации НОВЫХ сессий. Уже существующие/незавершённые сессии не затрагиваются (их открытые вопросы, если есть, уже в БД; если нет — ведут себя как «только закрытые», см. предыдущий анализ совместимости).

## Тестирование

- `open_seed.py`: `SEED_OPEN_QUESTIONS` непуст, каждый элемент — валидный `OpenQuestion` (проходит `_check`), `stem` содержит маркеры «Задание:», «Фокус ответа:», «Критерии оценки:».
- `build_open_stem`: собирает текст с нужными блоками и переносами.
- `generate_open_questions`: парсит новые поля, собирает `stem` через общий шаблон (мок-клиент).
- `generator`: пул = seed + LLM; ровно 2 выбираются; детерминизм по session.id (один и тот же session.id → один и тот же выбор); при «упавшем» LLM (мок бросает) пул = seed и сессия получает 2 открытых; `generated_count` покрывает open seq.
- Скрытность rubric: regression — `rubric` не уходит клиенту (уже есть в `test_results_has_open_section`).
- Frontend: `QuestionCard` рендерит многострочный `stem` открытого вопроса; подсказка про лимит присутствует.

## Вне scope (YAGNI)

- Отдельная таблица/админка для seed-вопросов в БД.
- Отдельные колонки `answer_focus`/`criteria` (решено: вшиваем в stem).
- Жёсткий счётчик символов 2500 во фронте (мягкая подсказка достаточно).
- Дедуп по семантике между fixed и LLM (только по индексу пула).
